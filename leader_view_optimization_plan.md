# 智能巡检大屏（领导视角）优化方案（代码检查版）

## 1. 现状检查结论（基于当前代码）

### 1.1 指标口径问题
- `renderStats` 中存在随机值和硬编码，导致领导看板数据不稳定、不可追溯：
  - 当日里程使用随机数：`Math.random()`（`app.js:1157`）。
  - 累计巡检时长硬编码：`12480`（`app.js:1169`）。
  - 顶部 KPI 含“效率提升”等推导型展示，缺少真实对比基线（`app.js:1180-1187`）。
- 码头/停靠点统计与数据源不一致：
  - 点位总数被写死为 `DATA.docks.length + 10`（`app.js:781`）。
  - 空闲/占用/故障写死为 `8/3/1`（`app.js:784-786`）。
- 告警分类占比条写死为 `50/30/20`（`app.js:748-752`），与真实告警结构脱节。

### 1.2 领导关注指标覆盖不足
- 当前“任务相关”仅有执行任务数和平均进度，缺少你提出的关键口径：
  - 任务执行总数（累计）
  - 执行次数（按任务重复执行）
  - 进行中总数
  - 失败总数
- 缺少“巡检点层”的全局指标：
  - 巡检点总数、已巡检点数、覆盖率、异常点数。
- 缺少环境指标聚合视图（当前只是点位明细，不是领导摘要）。

### 1.3 布局层面问题
- 虽然已有左右布局（`index.html` 左右 `panel`），但内容偏“操作台”而非“管理驾驶舱”：
  - 左侧更偏执行列表
  - 右侧更偏告警明细
  - 缺少“左经营、右风险/健康”的信息分层
- 顶部与中部指标重复，且缺少统一维度（时间范围、口径说明）。

## 2. 建议目标（领导视角）

把页面升级为“结果导向 + 风险导向”的双栏驾驶舱：
- 左侧：巡检计划与机器人效率（经营执行面）
- 右侧：巡检点健康与环境风险（安全质量面）
- 顶部：6 个核心总览 KPI（可解释、可追溯、可下钻）

## 3. 指标体系（建议落地口径）

## 3.1 顶部总览 KPI（全局）
1. 计划任务总数  
口径：时间窗内纳入计划的任务数量（不含草稿）
2. 执行总数  
口径：时间窗内任务实例执行次数总和（支持同一任务多次执行）
3. 进行中总数  
口径：状态为 `running` 的任务实例数
4. 失败总数  
口径：状态为 `failed` 的任务实例数
5. 巡检点覆盖率  
口径：已巡检点数 / 应巡检点总数
6. 风险闭环率  
口径：已闭环告警数 / 告警总数

## 3.2 左侧指标（计划 + 机器人）
- 计划执行看板：
  - 今日计划数、已执行次数、完成率、超时任务数
- 机器人运营看板：
  - 在线数、执行中、充电中、故障数、平均电量、利用率
- 趋势区（最近 7/30 天）：
  - 每日执行次数趋势
  - 失败率趋势

## 3.3 右侧指标（巡检点 + 环境）
- 巡检点看板：
  - 巡检点总数（你特别关注）
  - 已巡检点数
  - 异常点数（warn/danger）
  - 漏检点数
- 环境指标总览（按最新值聚合）：
  - O2、CH4、CO、H2S、TVOC、Noise 的超阈点位数
  - 环境健康指数（0-100，按超阈权重扣分）

## 4. 数据模型改造建议（先补口径，再改 UI）

当前 `DATA.tasks` 更像“任务定义”，缺少“执行实例”。建议新增：

```js
taskExecutions: [
  {
    execId: 'E-20260402-001',
    taskId: 'T-01',
    robotId: 'R-07',
    status: 'running', // pending/running/completed/failed/cancelled
    startedAt: '2026-04-02T09:50:00+08:00',
    endedAt: null,
    inspectedPoints: 12,
    totalPoints: 20,
    failedReason: null
  }
]
```

同时补充：
- `inspectionPoints` 全量主数据（当前分散在 `taskHierarchy`）
- `envThresholds` 环境阈值配置（便于统一判断超阈）

## 5. 页面布局优化（左右合理分布）

## 5.1 建议结构
- 顶部：`Leadership KPI Row`（6 卡，固定口径）
- 左栏（40%）：
  - `计划执行总览`
  - `机器人运营效率`
  - `执行趋势`
- 右栏（60%）：
  - `巡检点覆盖与异常`
  - `环境健康指数 + 指标超阈分布`
  - `高风险点 TopN`

说明：地图保留在中景层，侧栏上移为“摘要优先”，明细通过点击下钻弹出。

## 5.2 交互策略
- 所有 KPI 支持点击下钻：
  - 点击“失败总数” -> 失败任务列表
  - 点击“异常点数” -> 地图高亮对应巡检点
- 增加时间窗切换：`今日 / 近7天 / 近30天`
- 所有卡片显示口径提示（hover 或小问号）

## 6. 代码改造步骤（按风险从低到高）

### 第一步：去硬编码和随机值（1 天）
- 改 `app.js`：
  - 替换随机里程、写死停靠点统计、写死告警占比为真实计算。
  - 抽出统一函数 `computeDashboardMetrics(DATA, state)`。

### 第二步：建立领导指标计算层（1-2 天）
- 新增 `metrics.js`（或在 `app.js` 内先模块化）：
  - `computeTaskMetrics`
  - `computeRobotMetrics`
  - `computeInspectionPointMetrics`
  - `computeEnvironmentMetrics`
- 让 `renderStats / renderDocks / renderAlerts` 全部只消费计算结果，不直接拼业务逻辑。

### 第三步：重排左右布局与卡片（1-2 天）
- 改 `index.html`：新增领导卡片容器（计划、机器人、巡检点、环境）。
- 改 `main.css`：调整双栏比例与卡片优先级，确保 1920x1080 主展示效果。

### 第四步：补任务执行实例模型（2-3 天）
- 在 mock 数据先加入 `taskExecutions`，实现“执行多次、执行总数、失败总数”真实口径。
- 后续接后端时按同口径无缝替换。

## 7. 首批可直接上线的指标（不依赖后端新增）

基于当前静态数据，已可稳定输出：
- 机器人总数：5
- 任务池总数：4（`T-01~T-04`）
- 进行中任务：2
- 巡检点总数：4（`IP-E3/E5/B1/B3`）
- 告警总数：3
- 已处理告警：0

说明：  
“执行总数/失败总数/重复执行次数”需要 `taskExecutions` 这类执行实例数据，当前模型无法真实表达，建议作为第二阶段必须项。

## 8. 验收标准（领导视角）

1. 任意指标刷新后数值稳定，不再出现随机跳变。  
2. 每个指标可追溯到明确公式和数据源。  
3. 左侧回答“干了多少、效率如何”，右侧回答“风险在哪里、环境是否健康”。  
4. 领导在 15 秒内能回答四个问题：  
   - 今天计划做了多少？  
   - 正在执行多少？失败多少？  
   - 巡检点覆盖是否达标？  
   - 环境风险是否上升？

## 9. 建议你下一步优先级

1. 先做“去硬编码 + 统一指标计算层”（收益最高，改动最小）。  
2. 再补 `taskExecutions`，把“执行总数/失败总数”做实。  
3. 最后做布局重排和视觉强化，形成完整领导驾驶舱。
